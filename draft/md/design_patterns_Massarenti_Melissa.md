# Second step analysis - Prettier v3.8.2

## Pattern Used

### 1. Strategy pattern (behavioural)
Prettier's architecture is built to be extensible to different programming languages without modifying the central formatting engine. This separation is achieved through the Strategy Pattern.

Gli strumenti utilizzati sono `find` e `ls -F`.
Il comando impiegato per estrarre questo pattern: 
```bash
find src/language-* -name "parser-*.js"
```

I risultati ottenuti sono i seguenti:
```bash

src/language-graphql/parser-graphql.js
src/language-handlebars/parser-glimmer.js
src/language-json/parser-json.js
src/language-markdown/parser-markdown.js
src/language-yaml/parser-yaml.js
```

### 2. Facade Pattern (structural)
Prettier is made up of numerous complex submodules (comment management, option resolution, cursors, etc.). To prevent the user from having to interact with all these components, Prettier uses the Facade Pattern to provide a simplified interface.

### 3. Visitor Pattern (behavioural)
Prettier must parse the parse tree (AST) generated by the various parsers for each language. To separate the "formatting" logic from the physical structure of the tree (the nodes), the Visitor Pattern is used. This allows you to add new printing algorithms without having to modify the classes representing the code nodes.

Lo strumento utilizzato è `grep`.
Il comando impiegato per estrarre questo pattern: 
```bash
grep -r "getVisitorKeys" src/language-js/ | head -n 5
```

### 4. Singleton (creational)
Prettier handles the search and loading of configuration files (.prettierrc). To avoid scanning your hard drive every time a file is formatted (which would kill performance), use a centralized state manager, often implemented as a logical cache-based singleton.

Lo strumento utilizzato è `grep`.
Il comando impiegato per estrarre questo pattern:

### 7. Interpreter Pattern (behavioural)
Even though it is not a classical language interpreter, Prettier's src/document/ module defines an intermediate language (Doc) that is "interpreted" to decide where to wrap. Fixes the "line breaking" problem. Prettier does not print the code directly, but generates an intermediate representation that is then interpreted to calculate the maximum line length.

Gli strumenti utilizzati sono `ls` e `tree`.
Il comando impiegato per estrarre questo pattern:

### Summary Table: Design Patterns Assessment (Prettier v3.8.2)

| Pattern | Which classes play which role? | Why is the pattern used? (Problem solved) | Is there an alternative? (Pros & Cons) |
| :--- | :--- | :--- | :--- |
| **Strategy** (Behavioral) | **Context:** `src/main/core.js`<br>**Concrete Strategy:** `src/language-*/index.js` | **Multi-language support:** Encapsulates language-specific parsing and printing logic, making the core independent of specific syntax rules. | **Monolithic Switch-Case:**<br>+ Simpler for very small projects.<br>- Violates Open/Closed Principle; extremely difficult to maintain. |
| **Facade** (Structural) | **Facade:** `src/index.js`<br>**Subsystems:** `comments.js`, `options.js`, `parser.js`. | **API Complexity:** Provides a simplified unified interface (`format()`) to a complex system of internal modules, hiding details from the user. | **Direct Module Exposure:**<br>+ Maximum control for power users.<br>- High barrier to entry; unstable API (internal changes break user code). |
| **Visitor** (Behavioral) | **Visitor:** `printer.js` and `print/` folder.<br>**Elements:** AST nodes (from external parsers). | **Algorithm/Data Separation:** Allows defining "printing" operations without modifying node classes that often belong to third-party libraries. | **Internal Print Methods:**<br>+ More intuitive for simple structures.<br>- Tight coupling; impossible with third-party AST nodes. |
| **Singleton / Cache** (Creational) | **Manager:** `config/prettier-config/index.js`<br>**State:** `cache` object. | **I/O Performance:** Prevents redundant disk scans for configuration files by maintaining a single shared instance of config data. | **Dependency Injection:**<br>+ Improved testability (no global state).<br>- "Prop drilling": requires passing config through many nested calls. |
| **Composite** (Structural) | **Component:** AST Node interface.<br>**Leaf:** Simple nodes.<br>**Composite:** Block/Array statements. | **Recursive Tree Handling:** Allows the printer to treat individual nodes and nested structures (collections) uniformly during traversal. | **Manual Type Checking:**<br>+ No need for recursive abstractions.<br>- Complex and error-prone logic in every single print function. |
| **Command** (Behavioral) | **Invoker:** `src/cli/index.js`<br>**Commands:** `src/cli/options/`<br>**Receiver:** `src/main/core.js` | **CLI Decoupling:** Separates the terminal interface and command-line flags from the actual formatting execution logic. | **Hardcoded CLI Logic:**<br>+ Faster execution for tiny scripts.<br>- Low modularity; testing CLI logic independently is nearly impossible. |
| **Interpreter** (Behavioral) | **Expressions:** `src/document/` (Doc IR types).<br>**Interpreter:** Doc Printer (layout engine). | **Complex Layout Rules:** Evaluates an Intermediate Representation (Doc) to calculate optimal line breaks and indentation dynamically. | **Greedy String Concatenation:**<br>+ Much faster and simpler logic.<br>- Poor quality; unable to handle "break-if-too-long" formatting rules. |
